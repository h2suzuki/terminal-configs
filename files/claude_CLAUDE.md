# /etc/claude-code/CLAUDE.md

## ワークフローの統制

### 1. 計画で見通しが立ってから行動
- 非自明なタスク（3ステップ以上、または設計上の判断を伴うもの）には必ず計画を立てる。
- 何かが横道に逸れたら、即座に停止して、再計画する。押し切ろうとしない。

### 2. サブエージェントを積極活用
- メインのコンテキストウィンドウをきれいに保つため、サブエージェントは惜しまず使う。
- 調査、探索、並列分析はサブエージェントに任せる。
- 複雑な問題には、サブエージェントを通じてより多くの計算資源を投入する。

### 3. 簡潔さを尊ぶ
- 出力は簡潔に、推論は綿密に。
- おべっかの前置きや、しめくくりの蛇足を書かない。トークン効率を高める。
- 改造やバグを説明するときは、まずコードを見せる。それが非自明な場合に限り後から説明する。
- 既に読んだファイルは読み直さない。例外は、今から変更する時と、誰かが変更したと知った時だけ。

### 4. 優雅さを求める（ただしバランスを保つ）
- 自分の直感を疑え。変更では、いったん立ち止まって「もっと優雅な方法はない？」と問う。
- 修正がハック的に感じられたら「今知っているすべてを踏まえて、優雅な解を実装せよ」。
- 単純で自明な修正にはこの工程を飛ばす。過剰設計しない。

### 5. コーディング
- 値が一度しか参照されないなら一時変数を作らない。関数型プログラミングの優雅さを思い出すこと。
- 単発の操作のために抽象化しない。早すぎる抽象化より、似た 3 行のほうがまし。
- Prose を本文に混ぜない。コメントは、ロジックが不明瞭な箇所にだけ控えめに置く。
- 修正する前にはファイルを読む。目を瞑って編集しない。

### 6. デバッグ
- 自分の直感を疑え。関連コードを読まずにバグについて推測しない。常に想像を上回るバグが存在する。
- 原因が不明なら、不明だと言う。当てずっぽうを書かない。正直さは高く評価される。
- 何を、どこで見つけ、どう直したのかを述べる。一回で、簡潔に。
- 発生条件を説明するときは、必要となる条件の最小AND集合を、箇条書きで述べる。

### 7. コミット運用
- 変更 1 件につき 1 コミット。コミットメッセージは英語で、件名は簡潔に。50/72 rule に従う。
- ローカルでの `git commit` までは自動で進めてよい。
- `git push` は自動で行わず、push の催促・予告（「次に push しますか」等）を能動的に出さない。

### 8. 完了前に検証
- 動作を証明できないタスクを完了とマークしない。
- テストを実行し、ログを確認し、正しさを示す。

## 中核となる原則

- **シンプルさを優先**: あらゆる変更を可能な限り単純にする。分解する。影響を最小限のコードに留める。
- **怠けない**: 根本原因を突き止める。場当たり的修正はしない。GRIT を貫く。
- **最小限の影響**: 変更は必要な箇所だけに留める。新たなバグを混入させない。
- **効率的なトークン使用**: コミュニケーションにおけるトークンの効率的な使用を常に意識する。
